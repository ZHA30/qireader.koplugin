# QiReader Plugin Logic

本文是交接备忘，只保留会影响后续实现路径的关键约束。

## 入口边界

插件入口在 FileManager 主菜单的 `search` 分类中。打开后直接进入 `Subscriptions` 页面，不存在独立登录首页。

当前 UI 只稳定覆盖订阅域：订阅分组、订阅源、未读数、账号入口、未读过滤。文章列表和已读写回仍然是后续工作，不要反向约束当前订阅页实现。

## 协议假设

QiReader 没有面向插件的正式协议，当前实现基于 Web API 探测。新增接口前先补 `docs/api/` 记录，再接客户端代码。

订阅页当前依赖两个数据面：

- `/subscriptions`
- `/markers/unread/counts`

登录态基于服务端 Cookie。插件保存 `cookie`、`user`、`subscriptions_version`，不保存密码。

`401` 的处理约定已经固定：清空本地会话和当前分组数据，回到订阅页空态，再由 `Account` 入口重新登录。不要再引入“回到登录页”的旧模型。

## 分组语义

`!all` 是特殊集合。当前 UI 把它显示为顶部的 `全部`，用来承接“所有订阅”入口；它不算业务分组。

“未分组订阅源”的判断规则也已固定：只出现在 `!all` 中的订阅源，视为未分组；出现在其他分类中的订阅源，归入对应业务分组。

分组顺序遵循服务端返回顺序。分组内订阅源和未分组订阅源按标题本地排序。

`show_unread_only` 是纯视图过滤，不改变服务端数据，也不改变本地分组关系。

## UI 约定

订阅页继续使用 KOReader 原生 `Menu`，不自建整页列表框架。

顶部左侧按钮打开锚定二级菜单；二级菜单只保留两个入口：

- `Account`
- `Unread only: On/Off`

`Account` 在未登录时走原生 `MultiInputDialog` 登录，在已登录时走登出确认框。

列表交互约定如下：

- `全部` 没有折叠按钮，点击整行直接进入目标
- 业务分组左侧 `state` 区域负责折叠/展开
- 业务分组标题点击保留给后续文章列表入口

列表样式约定如下：

- 分组项粗体，已读时灰显
- 订阅源常规字重，已读时灰显
- `mandatory` 列显示未读数

## 状态与清理

控制器持有当前 UI 和临时状态，包括 `menu`、`active_dialog`、`login_dialog`、分组数据和展开状态。

顶层状态保持在 `closed`、`loading`、`groups`、`error` 这几个值；登录流程是弹窗，不单独占一个页面状态。

刷新订阅页应继续走 `Menu:switchItemTable()`，不要通过销毁并重建顶层页面刷新。

所有关闭路径都要统一回收控制器持有的菜单和弹窗引用。网络请求返回后先检查控制器是否已经关闭，避免旧回调刷新失效 UI。

## 后续边界

进入文章域之前，先确认文章列表接口、分页参数、已读/未读标记接口，以及服务端排序与同步语义。

在这些语义未确认前，不要提前设计本地缓存、离线推导或复杂同步模型。
