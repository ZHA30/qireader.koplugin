# QiReader Plugin Logic

本文记录影响后续开发走向的背景、原因和关键取舍。它不是开发日志，也不列已完成事项。

## 目标边界

QiReader 插件的首要目标是在 KOReader 内读取并操作 QiReader 账户数据。当前入口以“订阅分组”为首页，因为订阅结构是后续文章列表、阅读入口、未读状态和同步行为的上游索引。

插件应优先贴合 KOReader 的原生交互模型。除非原生组件无法表达关键交互语义，否则不要自创页面框架、底部栏、分页控件或通用列表容器。这样可以减少生命周期、焦点、按键、分页、刷新和设备差异带来的维护成本。

## API 与会话模型

QiReader Web API 没有公开的稳定插件协议，当前实现基于对 Web 端接口的探测结果。后续新增接口时，应先补充 `docs/api/` 下的 OpenAPI 和探测记录，再接入客户端代码。

登录态依赖服务端 Cookie。插件保存的是会话 Cookie 和用户信息，不保存密码。密码只在登录流程的内存字段中短暂存在，登录成功后清空。遇到 `401` 应清除本地会话并回到登录页，而不是静默重试旧 Cookie。

订阅首页依赖两个数据面：

- `/subscriptions` 提供订阅、分类和订阅到分类的关系。
- `/markers/unread/counts` 提供各订阅源未读数。

未读数是 UI 状态的重要输入，不只是附加显示。后续如果要增加文章列表、分组统计或同步状态，应继续以服务端未读计数为准，避免在本地推导长期状态。

## 分组语义

QiReader 的分类数据包含特殊分组 `!all`。它不能作为业务分组显示，但它会参与“所有订阅”的关系集合。判断“独立订阅源”时，只应把非 `!all` 分组中的关系视为已分组；不在任何业务分组中的订阅源才显示在分组列表之后。

分组顺序遵循服务端分类顺序。分组内订阅源和独立订阅源按标题排序，便于稳定浏览。后续如果要引入自定义排序，应明确这是插件本地排序，而不是 QiReader 服务端排序。

## UI 取舍

订阅首页应使用 KOReader 原生 `Menu`，不是独立自绘列表。原因如下：

- `Menu` 已提供标题栏、分页、返回、按键、焦点和刷新策略。
- Kindle/KOReader 的输入路径复杂，复用原生组件比维护自定义 `InputContainer` 更稳。
- 插件当前只需要扩展行内语义，不需要重建整个菜单系统。

订阅行使用 `Menu` 的 `state` 列表达折叠/展开符号，使用 `mandatory` 列显示未读数。未读数不加括号，靠右显示。菜单项之间的行线使用原生 `line_color` 控制。

折叠/展开只绑定到左侧 state 区域。标题区域保留给后续文章列表页。当前标题点击只记录目标，不应打开临时详情页或调试页。

未读和已读的视觉区分通过 `Menu` item 的 `bold` 字段表达：

- 未读数大于 `0` 的分组或订阅源使用加粗。
- 未读数为 `0` 的订阅源使用常规字重。

不要为了这些效果新建自定义列表 widget；优先使用 `Menu` 已有字段，包括 `state`、`mandatory`、`bold`、`line_color`、`single_line`、`align_baselines` 和原生分页。

## 状态与生命周期

控制器是插件 UI 状态的归属点。可见状态应保持少量且明确：登录、加载、分组首页、错误、关闭。网络请求返回后必须检查当前状态，避免关闭后的 UI 被旧请求刷新。

`Controller:close()` 负责关闭输入框和菜单引用。后续新增文章列表、设置页或确认框时，也应把实例引用放在控制器中，并在关闭路径统一清理。

动态菜单刷新应使用原生 `Menu:switchItemTable()`，而不是关闭再新建顶层 UI。这样能保留原生菜单分页和输入语义，也降低窗口生命周期问题。

## 后续开发方向

文章列表页应从“分组标题/订阅源标题点击”进入。进入时需要区分目标类型：

- 分组：按分类聚合文章。
- 订阅源：按单个 subscription 过滤文章。
- 独立订阅源：与普通订阅源相同，只是不属于业务分组。

文章列表实现前，应先继续探测文章接口、分页参数、已读/未读标记接口和服务端排序规则。不要先做本地缓存模型，除非已经明确服务端分页和同步行为。

后续任何 UI 增强都应先判断是否能用 KOReader 原生组件表达。只有在原生 `Menu`、`InputDialog`、`ButtonDialog` 等组件无法表达关键交互时，才考虑新增局部 widget；即便新增，也应只覆盖必要的行或局部区域，不替换整页菜单框架。
